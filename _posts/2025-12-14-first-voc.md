## 개요

iOS 출시 소식을 공유한 이후,
Android 때와는 비교가 되지 않을 정도로
다운로드 수가 빠르게 증가했습니다.

기쁜 마음도 컸지만 동시에 걱정도 따라왔습니다.
혼자 개발한 서비스가
과연 이 속도를 감당할 수 있을지에 대한 부담이었습니다.

사용자들이 앱을 설치했다는 것은
그만큼 신뢰를 보내주셨다는 의미라고 생각합니다.
그래서 출시 후 며칠 동안은
틈틈이 테스트를 진행하며 긴장된 상태로 지켜보고 있었습니다.

<br/>


## 첫 번째 VOC: 달력 UI

출시 3일째 되던 날, 드디어 **첫 VOC**가 들어왔습니다!

### 요청 내용

![Image](https://github.com/user-attachments/assets/f7bc06a5-9d7a-46a5-9b32-702598e6cdd4?raw=true)

바텀시트 형태로 제공되는
달력 UI에 대한 개선 요청이었습니다.

### 문제 상황

월마다 일수가 다른데, (28~31일) 그에 따라 바텀시트 높이가 달라진다. 월을 변경할 때마다 UI가 위아래로 움직인다.

### 사용자 제안

> “바텀시트 높이를 고정시키면 어떨까요?”

개발 과정에서는 크게 인식하지 못했던 부분이었지만
실제 사용자의 시선에서는 바로 불편함으로 느껴졌던 지점이었습니다.

### 반영
구조적으로 큰 수정이 필요하지 않은 부분이었기 때문에
확인 후 바로 수정 작업을 진행했습니다.

### 수정 내용

- 월과 관계없이 동일한 높이 유지
- UI 이동 최소화

작은 변화였지만
사용 경험을 훨씬 안정적으로 만들 수 있었습니다.

<br/>

## 두 번째 VOC: 반복 기능 오류

### 버그 리포트

두 번째 VOC는
반복 기능이 정상적으로 동작하지 않는다는 내용이었습니다.

반복 기능은
OrbitMe의 핵심 기능 중 하나이기 때문에
조금 더 긴장하게 되었던 이슈였습니다.


### 재현 시도

즉시 내부 테스트를 진행했지만
같은 현상을 재현할 수는 없었습니다.

### 사용자 데이터 확인

재현이 되지 않는 경우에는
실제 사용자 환경을 확인하는 수밖에 없었습니다.

#### 대응 과정

1. 사용자 닉네임 요청
2. 서버에서 해당 계정 데이터 확인
3. 반복 설정 상태 점검
4. 동일 조건으로 재시도

### 결과

이후 사용자로부터 다시 연락을 받았습니다.

> “지금은 정상적으로 작동합니다.”

일시적인 네트워크 이슈나
앱 재실행으로 해결된 캐시 문제로 추정되지만
명확한 원인을 특정하지 못한 점은 아쉬움으로 남았습니다.

<br/>

## VOC 대응을 통해 느낀 점

### 빠른 대응의 장점

혼자 개발하는 구조이기 때문에
의사결정과 수정 속도는 비교적 빠른 편입니다.

- 사용자와 직접 소통 가능
- 피드백 즉시 반영
- 수정 → 배포 사이클이 짧음

반면,
모든 상황을 실시간으로 대응하기는 어렵고
재현이 어려운 버그에 대한 한계도 분명히 느꼈습니다.

<br/>

## 배운 점

### 1. 사용자는 디테일을 놓치지 않는다
개발자 입장에서는
“기능이 정상적으로 동작하는지”에 집중하게 되지만
사용자는 “사용하면서 느껴지는 불편함”에 더 민감합니다.

작은 흔들림이나 변화도
사용자 경험에는 크게 다가올 수 있다는 것을 다시 느꼈습니다.

### 2. 재현되지 않는 버그를 대비해야 한다

모든 버그가
개발 환경에서 그대로 재현되지는 않습니다.

앞으로는

- 로깅 강화
- 사용자 행동 추적 보완
- 문제 상황을 빠르게 파악할 수 있는 구조

이런 부분들을 조금씩 보완해 나가야겠다고 생각했습니다.

<br/>

## 마무리

출시 3일 만에 받은 첫 VOC는
조금은 긴장되었지만, 동시에 반가운 신호이기도 했습니다.

누군가 실제로 앱을 사용하고 있고
더 나은 경험을 기대하고 있다는 증거였기 때문입니다.

앞으로도 사용자 피드백을 가장 중요한 기준으로 삼아
조금씩, 하지만 꾸준히 OrbitMe를 개선해 나가려 합니다.
