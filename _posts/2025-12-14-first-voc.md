## 개요

iOS 출시 글을 올리자마자 안드로이드와는 비교도 안 될 정도로 **다운로드 수가 빠르게 증가**했습니다!

**감정의 교차:**
- 기쁨 반: 많은 분들이 관심을 가져주셔서
- 걱정 반: 혼자 개발한 서비스가 안정적일까?

사용자들은 저희를 믿고 설치해주시는 만큼,
**안정적이고 신뢰할 수 있는 서비스**를 제공해야 한다는 책임감이 들었습니다.

그래서 틈틈이 테스트를 진행하며 긴장감 속에서 출시 3일을 보냈습니다. 🥹

<br/>

## 첫 번째 VOC: 달력 UI 개선

출시 3일째 되던 날, 드디어 **첫 VOC**가 들어왔습니다!


### 요청 사항

![Image](https://github.com/user-attachments/assets/f7bc06a5-9d7a-46a5-9b32-702598e6cdd4?raw=true)

**바텀시트 달력 UI**에서 개선 요청이 들어왔습니다.

**문제**
- 각 월마다 일수가 다름 (28일 ~ 31일)
- 바텀시트 높이가 월마다 달라짐
- 달을 바꿀 때마다 UI가 왔다 갔다 함
- 사용자 경험 저하

### 사용자 제안

> "바텀시트 높이를 고정시키면 어떨까요?"

정말 좋은 제안이었습니다!
저는 개발하면서 이 부분을 미처 신경 쓰지 못했는데,
실제 사용자의 관점에서 바로 발견해주셨습니다.

### 즉시 수정

바로 수정이 가능한 부분이어서 즉시 대응했습니다.

**수정 내용**
- 모든 월에서 동일한 UI 제공

## 두 번째 VOC: 반복 기능 오류

### 버그 리포트

두 번째 VOC는 **반복 기능이 작동하지 않는다**는 내용이었습니다.

이건 심각한 문제였습니다.
반복 기능은 OrbitMe의 핵심 기능 중 하나입니다.

### 재현 시도

즉시 내부 테스트를 진행했습니다.

하지만... **재현이 불가능**했습니다.

### 사용자 직접 대응

재현이 안 되는 상황에서는 **사용자의 실제 데이터**를 확인해야 했습니다.

**대응 과정**
1. 사용자에게 닉네임 요청
2. 서버에서 해당 유저의 데이터 조회
3. 반복 설정 확인
4. 재현 시도

### 해결

다행히 사용자분께서 다시 연락을 주셨습니다.

> "지금은 정상적으로 작동합니다!"

**추정:**
- 일시적인 네트워크 문제
- 또는 앱 재시작으로 해결된 캐시 이슈

완전히 원인을 파악하지는 못했지만,
무사히(?) 마무리되었습니다.

## VOC 대응 시스템

### 빠른 대응의 중요성

혼자 개발하는 만큼 **빠른 대응**이 장점이 될 수 있습니다.

**장점**
- 의사결정 빠름
- 즉시 수정 가능
- 직접 사용자와 소통

**단점**
- 24시간 대응 어려움
- 복잡한 버그는 시간 소요
- 스케일 한계

<br/>

## 배운 점

### 1. 사용자의 눈은 예리하다

**개발자 관점**
"기능이 작동하면 되지"

**사용자 관점**
"이 높이가 왔다 갔다 하는 게 불편해요"

개발하면서 놓친 부분을 사용자가 바로 발견합니다.

### 2. 재현 불가능한 버그의 어려움

모든 버그를 재현할 수 있는 건 아닙니다.

**대응 방법**
- 로깅 시스템 강화 필요
- 사용자 데이터 직접 확인
- 비슷한 상황 재현 시도
