## 개요

처음 온보딩에서는 **간단한 인사와 목표만 입력**을 받는 플로우였습니다.

하지만 베타테스트를 진행하면서 큰 문제를 발견했습니다.

<br/>

## 발견한 문제

### 사용자들이 앱을 이해하지 못함

베타 테스터분들이 **이 앱이 무엇인지, 어떻게 사용해야하는지**에 대해서 인지를 못하고 있었습니다.

**베타테스터 피드백:**
```
"온보딩 끝났는데 이제 뭐 하나요?"
"무슨 앱인가요?"
"카테고리는 왜 만드는 건가요?"
```

### 온보딩의 실패

**이전 온보딩:**
- 인사말 Page 3
- 목표 Page 1
- 끝

**결과:**
- 앱의 목적을 모름
- 다음에 뭘 해야 할지 모름
- 이탈률 높음

<br/>

## 고민의 시작

어떻게 해야 사용자들이 **온보딩 플로우만 경험해도 앱의 의도를 파악**할 수 있을지에 대해서 고민하였습니다.

**설명해야 할 것들:**
- OrbitMe가 무엇인지
- 우주 컨셉의 의미
- 카테고리와 목표의 관계
- 어떻게 사용하는지
- 주요 기능들

하지만 또 다른 걱정이 생겼습니다.

### 딜레마: 설명 vs 이탈

기능들을 다 설명하려면 플로우가 많아져야 하는데
유저들이 이를 참고 온보딩을 무사히 마칠 수 있는지에 대한 걱정도 있었습니다.

<br/>

## 24개의 화면

고민 끝에 결정했습니다.

실제로 현재 온보딩 플로우는 **약 24개의 UI**(말풍선 포함 갯수)가 그려졌습니다 ㅎ,,!
<br/>

**많다고요?**
네, 저희도 압니다...
하지만 이렇게 많은 대신에 **유저들이 지루하지 않게** 하기 위해 몰입감을 주는 방법을 고민했습니다.

<br/>

## 해결 방법: 게임 튜토리얼처럼

마치 **게임의 튜토리얼**처럼 만들기로 했습니다!


### 우리가 적용한 것

**1. 인터랙티브**
- 읽기만 하는 게 아니라 직접 해봄
- "여기를 눌러보세요" → 실제로 눌러야 다음으로
- 성공하면 애니메이션 효과

**2. 단계별 학습**
- 한 화면에 하나의 개념만
- 이전 개념 위에 새로운 개념 추가
- 점진적 복잡도 증가

**3. 비주얼 중심**
- 긴 글 대신 애니메이션

## 배운 점

### 1. 길어도 괜찮다

**잘못된 생각**
"온보딩은 짧을수록 좋다"

**올바른 생각**
"온보딩은 효과적일수록 좋다"

길어도 재밌고 유익하면 사용자들은 따라옵니다.

### 2. 설명보다 경험

긴 글로 설명하는 것보다
직접 해보게 하는 것이 훨씬 효과적입니다.
